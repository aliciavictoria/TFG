\chapter{Desarrollo del sistema}
\label{cap:desarrollo}

Este capítulo describe la implementación del sistema experimental utilizado para ejecutar y evaluar las distintas estrategias de prompting. En concreto, se explica la arquitectura en dos fases, el formato de entrada y salida, el diseño de los prompts, el extractor de traducciones, el proceso de construcción del conjunto de datos y los principales problemas técnicos encontrados durante el desarrollo.

% ─────────────────────────────────────────────────────────────
\section{Arquitectura general}
\label{sec:arquitectura-general}
% ─────────────────────────────────────────────────────────────

El sistema se implementó en Python e inicialmente se organizó con un script que directamente extrae el texto con las hipótesis de traducción que devuelve el modelo. Pero debido a que el modelo no siempre responde como le pedimos (en este caso, con una estructura JSON que divida las frases traducidas en bloques para que sean más sencillas de rescatar), algunas extracciones podían no ser del todo fiables. Finalmente, para poder verificar la respuesta completa del modelo, se terminaron implementando dos scripts independientes. Esta separación fue una de las decisiones más importantes del desarrollo, ya que permite distinguir claramente entre la captura de respuestas del modelo y el cálculo posterior de métricas.

\begin{description}
	\item[\texttt{experiment\_runner.py}] Gestiona la comunicación con las APIs, construye los prompts de cada estrategia y guarda en disco la respuesta completa del modelo antes de aplicar ningún procesamiento adicional. El resultado de esta fase es el log crudo del experimento.
	
	\item[\texttt{rescore\_from\_raw.py}] Lee los logs crudos, aplica el extractor de traducciones y calcula las métricas. Como no realiza llamadas a la API, puede ejecutarse tantas veces como sea necesario, y este punto es importante ya que de esta manera se podría corregir un error del extractor mientras se verificaba que los resultados son fiables, sin consumir cuota adicional.
\end{description}

En las primeras pruebas, algunos errores no procedían de la respuesta del modelo, sino de la forma en que el sistema extraía las traducciones finales. Si la extracción se hubiese realizado directamente durante la llamada a la API, habría sido necesario repetir muchas ejecuciones. Al conservar siempre la respuesta original, fue posible corregir el extractor y recalcular las métricas a partir de los mismos datos.

\subsection{Formato de entrada: el puzzle}
\label{subsec:formato-entrada}

Cada puzzle se almacena en un archivo JSON independiente. El archivo contiene dos listas diferenciadas: \texttt{train\_pairs}, que se incorporan al prompt como ejemplos de entrenamiento, y \texttt{test\_pairs}, que se reservan para la evaluación. Las traducciones correctas de los pares de test no se muestran nunca al modelo durante la inferencia.

\begin{lstlisting}[
style=PromptText,
caption={Estructura de un puzzle (ejemplo: Hankun \texttt{lang2en})},
label={lst:json-puzzle}
]
{
"id": "linguini_012018020100",
"language": "Hankun",
"language_family": "Sino-Tibetan",
"translation_direction": "lang2en",
"train_pairs": [
{ "source": "nɤ ʒip tuʔ ne", "target": "Did you sleep?" },
{ "source": "ati kəmə nirum lapkʰi tʰi ne",
"target": "Did he see us?" }
],
"test_pairs": [
{ "source": "nɤ ʒip ku ne", "target": "Do you sleep?" },
{ "source": "ati kəmə nirum lapkʰi ran ne",
"target": "Did he see us?" }
]
}
\end{lstlisting}

\subsection{Formato del log crudo}
\label{subsec:formato-raw-log}

Cada llamada al modelo queda registrada en un archivo \texttt{.jsonl}, con una línea JSON por llamada. El registro se realiza justo después de recibir la respuesta, antes de que actúe el extractor. De esta forma, cualquier fallo posterior en el procesamiento no provoca la pérdida del texto generado por el modelo y además, en caso de problema, permite ver el log crudo para extraer incluso las traducciones manualmente.

\begin{lstlisting}[
style=PromptText,
caption={Registro de una llamada al modelo en \texttt{raw\_logs/}
(respuesta abreviada)},
label={lst:raw-log}
]
{
"timestamp":  "2026-06-22T10:15:32",
"strategy":   "baseline",
"step":       1,
"prompt":     "You are an expert linguistic analyst...\n
--- TRAINING EXAMPLES ---\n
nɤ ʒip tuʔ ne → Did you sleep?\n
...\n
--- YOUR TASK ---\n
Translate into English:\n
1. nɤ ʒip ku ne\n...",
"response":   "DICTIONARY:\nŋa = I \nnɤ = you \n
ati = he/him\n...\n
GRAMMAR RULES:\n
1. Sentences end with 'ne' (question marker).\n
2. Past tense: suffix tuʔ/tʰi; present: ku/ran.\n...\n
TRANSLATIONS:\n
1. Do you sleep?\n
2. Did he see us?\n
3. Do they know you?\n
4. Do we beat them?\n
5. Did we know you?\n
6. Did we go?"
}
\end{lstlisting}

\subsection{Formato del JSON de métricas}
\label{subsec:formato-metricas}

La salida de \texttt{rescore\_from\_raw.py} contiene, para cada estrategia, los campos \texttt{primary\_metrics}, \texttt{best\_metrics} y la lista de pasos con sus métricas individuales. El campo \texttt{primary\_metrics} corresponde al resultado que se reporta en las tablas principales. En cambio, \texttt{best\_metrics} se conserva únicamente como información diagnóstica, especialmente en estrategias iterativas donde puede interesar saber si algún paso intermedio obtuvo mejor resultado que el paso final. Como veremos, a veces estos dos campos no coinciden debido a que el modelo pueda devaluar la respuesta en los últimos pasos.

El Código~\ref{lst:json-metricas} muestra un extracto real del puzzle de Hakhun con la estrategia M1, que además, destaca por haber sido la única salida perfecta de todo el experimento.

\begin{lstlisting}[
style=PromptText,
caption={Extracto del JSON de métricas para Hakhun M1
(GPT-OSS-120B). Los valores son los obtenidos
en el experimento, con normalización aplicada.},
label={lst:json-metricas}
]
{
"metadata": {
"puzzle_id": "linguini_012018020100",
"language":  "Hakhun",
"model_key": "gpt-oss-120b-openrouter",
"num_test_pairs": 6
},
"results": {
"baseline": {
"primary_metrics": {
"corpus_chrfpp":      100.0,
"corpus_bleu":        100.0,
"corpus_ter":         0.0,
"exact_match_corpus": 100.0,
"complete":           true,
"missing_indices":    [],
"num_evaluated":      6,
"num_expected":       6,
"hypotheses": [
"Do you sleep?",
"Did he see us?",
"Do they know you?",
"Do we beat them?",
"Did we know you?",
"Did we go?"
],
"references": [
"Do you sleep?",
"Did he see us?",
"Do they know you?",
"Do we beat them?",
"Did we know you?",
"Did we go?"
],
"sentence_chrfpp": [100.0, 100.0, 100.0, 100.0, 100.0, 100.0]
}
}
}
}
\end{lstlisting}

Este extracto permite ver cómo se compara cada hipótesis generada por el modelo con su referencia. Las seis frases obtienen chrF++=100 porque coinciden exactamente con la referencia tras la normalización. Cabe notar que las referencias originales del dataset Linguini incluyen marcadores de número gramatical \texttt{(sg)} y \texttt{(pl)} —por ejemplo, \texttt{you(sg)} o \texttt{you(pl)}— que son anotaciones propias del corpus y no forman parte del inglés natural. Dado que los modelos producen inglés sin estos marcadores (lo cual es correcto), incluirlos en la evaluación penalizaría artificialmente las traducciones correctas. Por decisión final, se implementó una función de normalización (\texttt{normalize\_reference}) que elimina estos marcadores de \textbf{ambos lados} —hipótesis y referencia— antes de calcular cualquier métrica, garantizando que la evaluación mida únicamente la calidad lingüística de la traducción.

Para estrategias iterativas como M2, el JSON incluye además la evolución paso a paso. El siguiente extracto muestra cómo crece la calidad de Hakhun a lo largo de los 9 pasos de la estrategia incremental:

\begin{lstlisting}[
style=PromptText,
caption={Evolución del chrF++ por paso en M2 para Hakhun (GPT-OSS-120B).},
label={lst:json-m2-steps}
]
"incremental_combinations": {
"primary_step":    9,
"primary_rule":    "last",
"primary_metrics": { "corpus_chrfpp": 91.7 },
"best_metrics":    { "corpus_chrfpp": 91.7 },
"steps": [
{ "step": 1, "metrics": { "corpus_chrfpp": 14.3 } },
{ "step": 2, "metrics": { "corpus_chrfpp": 20.0 } },
{ ...                                              },
{ "step": 9, "metrics": { "corpus_chrfpp": 91.7 } }
]
}
\end{lstlisting}

El campo \texttt{primary\_rule: "last"} indica que se reporta el último paso, es decir, el momento en que el modelo ya ha recibido todos los ejemplos de entrenamiento. No se reporta automáticamente el paso con mayor puntuación, ya que eso introduciría un sesgo favorable al utilizar las respuestas de test para escoger el mejor resultado, además de no ser un resultado real, ya que éste se debe extraer siempre de la conclusión final. Esta decisión se resume junto con el resto de decisiones técnicas en la Tabla~\ref{tab:decisiones-tecnicas}, situada en el final de este capítulo. El paso máximo se conserva en \texttt{best\_metrics} únicamente como diagnóstico de estabilidad.

% ─────────────────────────────────────────────────────────────
\section{Diseño de los prompts}
\label{sec:prompts}
% ─────────────────────────────────────────────────────────────

Una vez vistas, en el apartado de metodología todas las estrategias de prompting de forma teórica, procedemos a observar las plantillas de prompt utilizadas por el sistema experimental. En lugar de mostrar ejemplos simplificados, se incluyen las estructuras completas empleadas en el script de lanzamiento. Algunas partes se generan dinámicamente en función del puzzle, como los pares de entrenamiento, las frases de test, la dirección de traducción o el número exacto de respuestas esperadas.

\subsection{Instrucción metalingüística base}
\label{subsec:instruccion-base}

Todas las estrategias principales, de M1 a M5, comienzan con una misma instrucción metalingüística. Su objetivo es fijar el papel del modelo como analista lingüístico y recordarle que debe razonar únicamente a partir de los ejemplos proporcionados.

\begin{lstlisting}[
style=PromptText,
caption={Instrucción metalingüística base utilizada en M1--M5.},
label={lst:prompt-base}
]
You are an expert linguistic analyst performing inductive reasoning on an unknown language.
You have never seen this language before. You must reason exclusively from the provided examples using your meta-linguistic knowledge of how human languages work in general (word order typology, morphological patterns, agreement systems, affixation, etc.).

Core principles:
- Do NOT use any prior knowledge of this specific language or its family
- Treat every hypothesis as provisional and subject to revision
- Always show your reasoning explicitly before reaching any conclusion
- A wrong rule confidently stated is worse than uncertainty acknowledged
\end{lstlisting}

Los prompts se redactan en inglés porque es la lengua en la que los modelos suelen mostrar mejor rendimiento en tareas de razonamiento, y porque las traducciones de los puzzles tienen siempre el inglés como lengua origen o destino. Además, la instrucción de no utilizar conocimiento previo es importante porque algunos modelos pueden haber visto información sobre determinadas lenguas durante su preentrenamiento.

\subsection{Bloque dinámico de traducciones}
\label{subsec:formato-salida}

Todas las estrategias que producen traducciones utilizan un bloque final \texttt{TRANSLATIONS}. Este bloque se genera dinámicamente con el número real de frases de test del puzzle. Así se evita mostrar al modelo un ejemplo genérico con solo dos líneas cuando, en realidad, debe responder tres, cuatro o seis frases.

\begin{lstlisting}[
style=PromptText,
caption={Bloque dinámico de salida para $n$ frases de test.},
label={lst:prompt-translations-block}
]
TRANSLATIONS:
1. [translation only]
2. [translation only]
...
n. [translation only]
=== END ===

You MUST provide all n numbered lines above, from 1 to n, in that exact
order. Never skip a number: if you are unsure, give your best guess anyway.
Write nothing after === END ===.
\end{lstlisting}

El marcador \texttt{=== END ===} se añadió posteriormente para facilitar el bucle en el código de la extracción automática (como una marca de dónde debe terminar la sección de traducciones), ya que algunos modelos añadían explicaciones después de las traducciones, y esas notas también se interpretaban como respuestas finales.

\subsection{Prompt de M0: verificación previa}
\label{subsec:prompt-m0}

M0 no utiliza la instrucción metalingüística base. Su objetivo es comprobar si el modelo puede traducir una frase de test sin haber recibido ningún ejemplo de entrenamiento. A esto le llamamos la verificación de data contamination.

Para puzzles en dirección lengua desconocida $\rightarrow$ inglés se utiliza:

\begin{lstlisting}[
style=PromptText,
caption={Prompt de M0 para dirección lengua desconocida $\rightarrow$ inglés.},
label={lst:prompt-m0-lang2en}
]
Translate the following sentence into English. Give your best attempt even if you are not sure.

Sentence: {sentence}

Translation:
\end{lstlisting}

Para puzzles en dirección inglés $\rightarrow$ lengua desconocida, la plantilla cambia la lengua de destino:

\begin{lstlisting}[
style=PromptText,
caption={Prompt de M0 para dirección inglés $\rightarrow$ lengua desconocida.},
label={lst:prompt-m0-en2lang}
]
Translate the following sentence into {language}. Give your best attempt even if you are not sure.

Sentence: {sentence}

Translation:
\end{lstlisting}

En ambos casos, \texttt{\{sentence\}} se sustituye por la primera frase del conjunto de test para cada problema a analizar. La traducción generada se compara con la referencia mediante chrF++. Si el valor supera el umbral definido (en este caso es 20), el puzzle se marca para revisión manual. En el caso de las pruebas realizadas para este proyecto, todos los puzzles devolvieron un valor inferior al límite.

\subsection{Prompt de M1: baseline}
\label{subsec:prompt-m1}

M1 presenta todos los pares de entrenamiento en un único prompt. Es la estrategia más directa y sirve como punto de comparación para las demás metodologías.

\begin{lstlisting}[
style=PromptText,
caption={Plantilla completa del prompt de M1.},
label={lst:prompt-m1}
]
{METALINGUISTIC_INSTRUCTION}

--- TRAINING EXAMPLES ---
{examples}

--- YOUR TASK ---
Based on ALL examples above, write your inferred DICTIONARY and GRAMMAR RULES,
then {translation_instruction}:
{tests}

Format your response EXACTLY as:
DICTIONARY:
[word mappings, one per line]

GRAMMAR RULES:
[rules, one per line]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

En esta plantilla, \texttt{\{examples\}} contiene todos los pares de entrenamiento del puzzle en formato \texttt{fuente -> traducción}. El campo \texttt{\{tests\}} contiene las frases que el modelo debe traducir, numeradas en el mismo orden en que serán evaluadas. Por último, \texttt{\{TRANSLATIONS\_BLOCK\}} se sustituye por el bloque dinámico descrito en la Sección~\ref{subsec:formato-salida}.

\subsection{Prompt de M2: presentación incremental acumulativa}
\label{subsec:prompt-m2}

M2 utiliza un prompt distinto en cada paso. En lugar de mostrar todos los ejemplos desde el principio, introduce subconjuntos crecientes y proporciona al modelo la hipótesis acumulada del paso anterior. El objetivo es que el modelo revise su diccionario y sus reglas a medida que recibe nueva evidencia.

\begin{lstlisting}[
style=PromptText,
caption={Plantilla completa del prompt de M2.},
label={lst:prompt-m2}
]
{METALINGUISTIC_INSTRUCTION}

STEP {step}/{total} — New subset of examples:

--- YOUR CURRENT HYPOTHESIS (provisional — review critically) ---
DICTIONARY:
{current_dictionary}

GRAMMAR RULES:
{current_rules}

IMPORTANT: New examples may CONFIRM, REFINE or CONTRADICT entries above.
Delete proven-wrong entries; correct imprecise ones.

--- NEW EXAMPLES ---
{new_examples}

--- YOUR TASK (in this order) ---

STEP A · ANALYSIS
Identify morphemes in each new example. Check for contradictions with your
current hypothesis. Think aloud. Do NOT translate the test sentences yet.

STEP B · REVISED DICTIONARY
One entry per line: morpheme = meaning (notes)
Mark deleted entries: [DELETED: entry — reason]

STEP C · REVISED GRAMMAR RULES
Numbered list. Mark deleted rules: [DELETED: rule — reason]

STEP D · TRANSLATIONS
{translation_instruction} — write ONLY the translation on each line, nothing else:
{tests}

Format your response EXACTLY as:
ANALYSIS:
[reasoning]

DICTIONARY:
[entries]

GRAMMAR RULES:
[rules]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

En el primer paso, cuando todavía no existe conocimiento acumulado, el bloque de hipótesis previa se sustituye por una indicación de que no hay conocimiento anterior. En los pasos siguientes, \texttt{\{current\_dictionary\}} y \texttt{\{current\_rules\}} contienen el diccionario y las reglas extraídas del paso previo. Esta estructura obliga al modelo a tratar sus hipótesis como revisables, no como conclusiones definitivas.

\subsection{Prompt de M3: step-by-step lingüístico}
\label{subsec:prompt-m3}

M3 divide los ejemplos en cuatro etapas: léxico, fonología, morfosintaxis y sintaxis. Cada etapa utiliza el mismo formato general, pero cambia el foco lingüístico que se le indica al modelo.

\begin{lstlisting}[
style=PromptText,
caption={Plantilla completa del prompt de M3.},
label={lst:prompt-m3}
]
{METALINGUISTIC_INSTRUCTION}

STEP {step}/4 — Stage: {stage}
{stage_focus}

--- ACCUMULATED KNOWLEDGE ---
DICTIONARY:
{current_dictionary}

GRAMMAR RULES:
{current_rules}

--- EXAMPLES FOR THIS STAGE ---
{stage_examples}

--- YOUR TASK ---
1. Refine DICTIONARY with new vocabulary
2. Add/update GRAMMAR RULES for this stage
3. {translation_instruction} using ALL accumulated knowledge

Format your response EXACTLY as:
DICTIONARY:
[word mappings]

GRAMMAR RULES:
[rules]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

Los focos lingüísticos utilizados en cada etapa son los siguientes:

\begin{lstlisting}[
style=PromptText,
caption={Focos lingüísticos utilizados en M3.},
label={lst:prompt-m3-focus}
]
LEXICAL:
Focus on vocabulary and basic word order.

PHONOLOGY:
Focus on phonological patterns: vowel changes, consonant alternations, affixes, allomorphs.

MORPHOSYNTAX:
Focus on agreement, tense, number, person, gender markings.

SYNTAX:
Combine all rules: negation, questions, complex clauses.
\end{lstlisting}

Aunque el formato está inspirado en una progresión lingüística, la partición de ejemplos es heurística. Los puzzles de \textsc{Linguini} no indican qué frase ilustra cada fenómeno lingüístico ni separan explícitamente los ejemplos por vocabulario, morfología, sintaxis o traducción completa. Por este motivo, el sistema reparte los pares de entrenamiento según porcentajes aproximados: una primera parte se utiliza para identificar vocabulario básico, una segunda parte para observar posibles regularidades morfológicas, una tercera para formular reglas sintácticas o gramaticales más generales y, finalmente, el conjunto completo se usa para producir las traducciones de test.

Esta división no pretende afirmar que cada bloque contenga únicamente ese tipo de información, ya que en un puzzle real una misma frase puede aportar simultáneamente vocabulario, marcas morfológicas y orden sintáctico. Se trata más bien de una forma de guiar progresivamente al modelo, obligándolo a construir hipótesis parciales antes de llegar a la traducción final. Por tanto, M3 debe entenderse como una estrategia de estructuración del razonamiento, no como una segmentación lingüística exacta del dataset.

\subsection{Prompt de M4: inspiración humana}
\label{subsec:prompt-m4}

M4 añade antes del puzzle nuevo una resolución humana completa del puzzle de Hakhun. Esta resolución no se utiliza como fuente de vocabulario ni de reglas para la nueva lengua, sino como ejemplo del tipo de razonamiento que se quiere inducir.

\begin{lstlisting}[
style=PromptText,
caption={Cabecera del razonamiento humano utilizado en M4.},
label={lst:prompt-m4-human-header}
]
The following is a real human's reasoning while translating an unknown language puzzle.
Use it as inspiration for your OWN reasoning process on a DIFFERENT language below.
Do NOT copy vocabulary or rules — they belong to a completely different language.

--- HUMAN REASONING EXAMPLE (Hakhun language, Sino-Tibetan) ---
\end{lstlisting}

Después de esta cabecera, el sistema inserta el razonamiento humano completo frase a frase. El contenido incluye la identificación de \texttt{ne} como marcador interrogativo, el análisis de los pronombres, la deducción del sistema aspectual y la identificación de verbos como \texttt{lapkʰi}, \texttt{cʰam}, \texttt{lan}, \texttt{ka/kɤ} y \texttt{ʒip}. El bloque termina con un resumen de los patrones encontrados.

\begin{lstlisting}[
style=PromptText,
caption={Resumen final del razonamiento humano insertado en M4.},
label={lst:prompt-m4-human-summary}
]
Key patterns identified:
- Tense suffixes: tuʔ/tɤʔ/tʰɤ/tʰu/tʰi/tiʔ = past (Did); ku/ki/rɤ/ri/ran/kan = present (Do)
- Pronouns: ŋa=I, nɤ=you(sg), ati=he, nirum=we, nuʔrum=you(pl), tarum=they
- 'kəmə' = transitive object marker
- 'lapkʰi' = see, 'cʰam' = know, 'lan' = beat, 'ka/kɤ' = go, 'ʒip' = sleep
- 'ne' = sentence-final question marker
--- END OF HUMAN EXAMPLE ---
\end{lstlisting}

A continuación, el prompt presenta el nuevo puzzle y recuerda explícitamente que no se debe transferir contenido lingüístico del ejemplo de Hakhun:

\begin{lstlisting}[
style=PromptText,
caption={Plantilla completa del prompt de M4 tras insertar el razonamiento humano.},
label={lst:prompt-m4}
]
{METALINGUISTIC_INSTRUCTION}

{HUMAN_RESOLUTION_HAKHUN}

Now apply the same style of reasoning to a NEW and COMPLETELY DIFFERENT linguistic puzzle.
The language below is unrelated to the example above — do not transfer any vocabulary or rules.

--- TRAINING EXAMPLES (new language) ---
{examples}

--- YOUR TASK ---
Following the same step-by-step reasoning shown in the human example,
write your DICTIONARY and GRAMMAR RULES, then {translation_instruction}:
{tests}

Format your response EXACTLY as:
DICTIONARY:
[word mappings, one per line]

GRAMMAR RULES:
[rules, one per line]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

Por supuesto, cabe mencionar de nuevo que para evitar la fuga de información, M4 no se ejecuta sobre el puzzle de Hakhun.

\subsection{Prompt de M5: segundo revisor}
\label{subsec:prompt-m5}

M5 trabaja en dos turnos. El primer turno usa el mismo prompt que M1. El segundo turno devuelve al modelo su propia respuesta inicial y le pide que actúe como revisor.

\begin{lstlisting}[
style=PromptText,
caption={Prompt del turno 1 de M5.},
label={lst:prompt-m5-t1}
]
{METALINGUISTIC_INSTRUCTION}

--- TRAINING EXAMPLES ---
{examples}

--- YOUR TASK ---
Based on ALL examples above, write your inferred DICTIONARY and GRAMMAR RULES,
then {translation_instruction}:
{tests}

Format your response EXACTLY as:
DICTIONARY:
[word mappings, one per line]

GRAMMAR RULES:
[rules, one per line]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

El segundo turno incluye el diccionario, las reglas y las traducciones generadas en el primer turno. La instrucción de crítica obliga al modelo a comprobar sus propias respuestas frente a los ejemplos de entrenamiento.

\begin{lstlisting}[
style=PromptText,
caption={Prompt completo del turno 2 de M5.},
label={lst:prompt-m5-t2}
]
{METALINGUISTIC_INSTRUCTION}

You have already analyzed the following language and produced a first draft.
Now act as a careful reviewer of your own work.

--- TRAINING EXAMPLES (same as before) ---
{examples}

--- YOUR FIRST-DRAFT ANALYSIS ---
DICTIONARY:
{first_draft_dictionary}

GRAMMAR RULES:
{first_draft_rules}

FIRST-DRAFT TRANSLATIONS:
{first_draft_translations}

--- YOUR TASK ---
1. CRITIQUE: For each translation above, check whether it is consistent with your
dictionary and grammar rules. List specific contradictions or uncertainties.
Be precise: quote the rule or entry that is violated or missing.

2. REVISED DICTIONARY: Update entries if needed.

3. REVISED GRAMMAR RULES: Update rules if needed.

4. FINAL TRANSLATIONS: Produce corrected translations based on your critique.
If a translation was already correct, keep it unchanged.

CRITICAL: Your task is to {translation_instruction}. Every line in TRANSLATIONS must be
in {target_language}.
Do NOT produce text in any other language.

Format your response EXACTLY as:
CRITIQUE:
[numbered list of issues, or "No contradictions found."]

DICTIONARY:
[entries]

GRAMMAR RULES:
[rules]

{TRANSLATIONS_BLOCK}
\end{lstlisting}

El recordatorio \texttt{CRITICAL} se añadió a posteriori, ya que tras observar que, en algunos experimentos, los modelos cambiaban de dirección durante la autocorrección y producía texto en la lengua desconocida en lugar de traducir al inglés. También se mantiene el bloque dinámico de traducciones para asegurar que el segundo turno devuelva exactamente el mismo número de respuestas que el primero. Este quizás es el prompt más complejo y, por desgracia, el menos certero. Parece ser que el modelo no llega a comprender su función de crítico y, por consiguiente, suele empeorar la hipótesis inicial en todos los casos.

\section{Extracción automática de traducciones}
\label{sec:extractor-traducciones}
% ─────────────────────────────────────────────────────────────

Las respuestas del modelo no contienen únicamente traducciones. Normalmente incluyen también análisis morfológico, diccionario, reglas gramaticales y comentarios adicionales. Por ello, antes de calcular las métricas es necesario aislar únicamente las traducciones finales. Esta parte ha sido todo un reto, ya que primero había que averiguar la manera de pedirle al modelo el formato exacto de fichero que debe devolver y, posteriormente, rescatar las frases traducidas de forma automática para ahorrar tiempo, ya que la extracción manual sería poco práctica.

\subsection{Algoritmo de extracción}
\label{subsec:algoritmo-extractor}

El extractor sigue tres pasos principales:

\begin{enumerate}
	\item \textbf{Localización de la sección.} Se busca la última aparición de \texttt{TRANSLATIONS:} en la respuesta, no la primera. Esta decisión es necesaria porque estrategias como M5 pueden mencionar la palabra \textit{Translations} en la sección de crítica antes de llegar a la sección final de resultados.
	
	\item \textbf{Extracción indexada por número.} Dentro de esa sección se analizan líneas del tipo \texttt{N. texto}. El número escrito por el modelo determina la posición de la hipótesis. Así, si el modelo escribe \texttt{3. texto}, ese texto se asigna siempre a la tercera frase de test, aunque las líneas anteriores no hayan sido reconocidas. Este cambio se ha tenido que implementar debido a un problema de extracciones anteriores, ya que cuando se perdía la primera traducción, las siguientes se desplazaban y acababan comparándose con referencias incorrectas, por lo que las métricas calculadas no eran fiables.
	
	\item \textbf{Filtrado.} Los candidatos pasan por varios filtros para descartar entradas de diccionario, instrucciones de revisión y análisis metalingüístico. Por ejemplo, se eliminan líneas con patrones del tipo \texttt{palabra = significado}, instrucciones como \texttt{add} o \texttt{delete} acompañadas de términos como \texttt{rule} o \texttt{morpheme}, y explicaciones que contienen expresiones como \texttt{word order}, \texttt{suffix} o \texttt{allomorph}. Para ello lo que más ha sido de ayuda es la revisión manual de los logs crudos que devolvía el modelo, donde podía observar que filtrados son necesarios para la extracción de textos traducidos. No obstante, este enfoque tiene una limitación importante, y es que al depender de expresiones y formatos concretos, puede comportarse de forma distinta según el modelo utilizado y según cómo cada modelo organice su respuesta.
\end{enumerate}

\subsection{Gestión de respuestas incompletas}
\label{subsec:respuestas-incompletas}

Cuando el extractor no encuentra una traducción para una posición, la deja como \texttt{None}. Durante el cálculo de métricas, esas posiciones se rellenan con cadenas vacías antes de calcular chrF++ a nivel de corpus. Así, una respuesta incompleta penaliza el resultado en lugar de evaluarse solo sobre las frases que el modelo sí respondió. Esta decisión se recoge también en la Tabla~\ref{tab:decisiones-tecnicas}.

Los campos \texttt{missing\_indices} y \texttt{complete} del JSON de métricas registran explícitamente qué posiciones se perdieron:

\begin{lstlisting}[
style=PromptText,
caption={Ejemplo de extracción incompleta en el JSON de métricas},
label={lst:extraccion-incompleta}
]
"primary_metrics": {
"corpus_chrfpp":   23.4,
"complete":        false,
"missing_indices": [0, 2],   // posiciones 1 y 3 sin traducción
"num_evaluated":   4,
"num_expected":    6,
"hypotheses": [
"",                          // pos. 0 vacía — penaliza en el corpus
"Did he see us?",
"",                          // pos. 2 vacía — penaliza en el corpus
"Do we beat them?",

"Did he see us?",
"Do they know you(pl)?",

"Did we know you(sg)?",

]
}

\subsection{Detección de bucles de repetición}
\label{subsec:bucles-repeticion}

Durante los experimentos se observó que algunos modelos, especialmente Llama-3.1-8B en prompts largos como M4 y M5, podían entrar en bucles de generación repetitiva. En esos casos, el modelo repetía entradas del diccionario hasta agotar el límite de tokens y no llegaba a producir la sección \texttt{TRANSLATIONS}. El código~\ref{lst:bucle-ejemplo} muestra un ejemplo real de un problema encontrado.

\begin{lstlisting}[
style=PromptText,
caption={Fragmento de respuesta con bucle de repetición
(Inuktitut, Llama-3.1-8B, M1)},
label={lst:bucle-ejemplo}
]
DICTIONARY:
...
197. The suffix "-ha" appears to indicate a third-person
singular subject performing an action.
198. The suffix "-ha" appears to indicate a third-person
singular subject performing an action.
199. The suffix "-ha" appears to indicate a third-person
singular subject...
[fin de tokens — nunca se genera TRANSLATIONS]
\end{lstlisting}

Como solución, se decidió que la función \texttt{call\_model} comprobara si la respuesta contenía seis o más líneas idénticas consecutivas. Si se detectaba un bucle, la llamada se repetía con una temperatura ligeramente superior, con el objetivo de introducir más variación y romper la repetición. En la configuración base, las llamadas se realizaron con \texttt{temperature=0.0}, \texttt{top\_p=1.0} y sin modificar \texttt{top\_k}, manteniendo el valor por defecto de cada proveedor. Cuando se detectaba un bucle, se repetía la llamada con \texttt{temperature=0.4}, manteniendo el resto de parámetros sin cambios. Se permitieron hasta cuatro intentos.

\subsection{Otras limitaciones}
\label{subsec:limitaciones-abiertas}

Algunas limitaciones no pueden resolverse por completo mediante cambios en el código. La dependencia de proveedores externos implica que el comportamiento de los modelos puede variar entre sesiones. Además, el control M0 solo permite detectar indicios de conocimiento previo, pero no demostrar con certeza que una lengua esté ausente de los datos de preentrenamiento.

También es importante tener en cuenta que las métricas automáticas no sustituyen al análisis manual. Dos traducciones formalmente distintas pueden ser lingüísticamente equivalentes, y una respuesta con chrF++ elevado puede contener un error gramatical relevante.
